# GitHub Actions CI/CD設計書

## 1. ワークフロー設計

### 1.1 ワークフロー一覧

| ワークフロー名 | ファイル名 | トリガー | 依存関係 | 用途 |
|--------------|-----------|---------|---------|------|
| CI Workflow | ci.yaml | push（main）、PR（main） | なし | 変更領域の検証、frontend本番ビルド、本番アプリ配布の起動 |
| Frontend Deploy | frontend-deploy.yaml | CI WorkflowまたはFrontend Rollbackから呼び出し | CI検証済みartifact | `frontend-dist` のS3配布・CloudFrontキャッシュ無効化 |
| Frontend Rollback | frontend-rollback.yaml | 手動 | 成功済みCI Workflow run | 保存済み`frontend-dist`を本番へ再配布 |
| Backend Deploy | backend-deploy.yaml | CI Workflowから呼び出し | deploy gate、必要時はfrontend検証 | DockerイメージのECR push・ECSデプロイ |
| Terraform Plan | terraform-plan.yaml | PR（main、terraform/**変更時） | なし | Terraformプラン実行・PRへのコメント |
| Terraform Apply | terraform-apply.yaml | push（main、terraform/**変更時）、手動 | なし | Terraformアプライ実行 |

### 1.2 ワークフロー実行フロー

#### 1.2.1 Pull Request作成時
```
PRを作成
    │
    ├─── frontend/**変更あり ──→ CI Workflow（format・lint・test・coverage・production build）
    │
    ├─── backend/**変更あり ───→ CI Workflow（バックエンドのテスト・リント）
    │
    └─── terraform/**変更あり ─→ Terraform Plan（plan結果をPRにコメント）

    ↓
レビュー・承認
    ↓
mainにマージ
```

#### 1.2.2 アプリケーションデプロイ時（mainへのマージ）

backend/、frontend/の両方に変更がある状態。
```
mainにマージ
    │
    ▼
┌────────────────────────────────────────────────────────────────────┐
│ Step 1: ci.yaml                                                     │
│ - frontend: format・lint・test・coverage・production build           │
│ - backend: ./gradlew check                                          │
│ - frontend build成功時: frontend-dist artifactを90日保持             │
└────────────────────────────────────────────────────────────────────┘
    │
    │ 変更対象の検証がすべて成功した場合のみ
    ▼
┌─────────────────────────────────┐
│ Step 2: backend-deploy.yaml     │
│ - Dockerイメージをビルド         │
│ - ECRにプッシュ                  │
│ - ECSへデプロイして安定化を待機   │
└─────────────────────────────────┘
    │
    │ backendデプロイ成功後
    ▼
┌─────────────────────────────────┐
│ Step 3: frontend-deploy.yaml    │
│ - frontend-distをダウンロード    │
│ - S3へ順次アップロード           │
│ - CloudFrontキャッシュ無効化     │
└─────────────────────────────────┘
```

#### 1.2.3 通常運用時（2回目以降、mainへのマージ）

変更されたアプリケーション領域に応じて必要な配布のみ実行する。
```
mainにマージ
    │
    ├─── backend/**のみ変更 ───→ backend-deploy.yamlのみ実行
    │
    └─── frontend/**のみ変更 ──→ frontend-deploy.yamlのみ実行
```

frontend/**とbackend/**の両方に変更がある場合、両方のCIが成功した後、backend、frontendの順にデプロイする。backendのCIまたはデプロイが失敗した場合はfrontendを公開しない。

terraform/**の変更は`terraform-apply.yaml`が独立して処理する。Terraform applyとアプリケーション配布の間に自動的な依存関係は設けていないため、新規環境作成やアプリ配布先を変更するインフラ更新では、必要なインフラ反映後にアプリケーション変更を反映する。

#### 1.2.4 フロントエンドロールバック時

```
Frontend Rollbackを手動実行（成功済みCI Workflow run_idを入力）
    │
    ▼
CI Workflowがmainへのpushで成功済みかつfrontend-distが利用可能であることを確認
    │
    ▼
保存済みfrontend-distをS3へ再配布し、CloudFrontキャッシュを無効化
```

### 1.3 GitHub Secretsに登録する値

| シークレット名 | 用途 |
|--------------|------|
| AWS_ACCOUNT_ID | AWSアカウントID（12桁） |
| TF_VAR_jwt_secret | JWT署名鍵 |
| TF_VAR_google_oauth_client_id | Google OAuthクライアントID |
| TF_VAR_google_oauth_client_secret | Google OAuthクライアントシークレット |
| TF_VAR_github_oauth_client_id | GitHub OAuthクライアントID |
| TF_VAR_github_oauth_client_secret | GitHub OAuthクライアントシークレット |
| TF_VAR_alert_email_address | CloudWatchアラート通知先メールアドレス |

### 1.4 OIDC連携

GitHub ActionsからAWSへの認証はOIDC連携を使用する。IAMロール`keirekipro-github-actions-role`の信頼ポリシーで、特定リポジトリからのAssumeRoleWithWebIdentityのみを許可する。

### 1.5 デプロイ戦略

| 対象 | 戦略 |
|------|------|
| フロントエンド | CIで一度だけbuildしたartifactをS3へ配布 + CloudFrontキャッシュ無効化。旧assetは保持し、成功済みartifactの再配布でrollback |
| バックエンド | ECSローリングアップデート（最小ヘルス率100%、最大率200%、回路ブレーカーによる自動rollback有効） |

frontendとbackendの同時変更では、APIの前後互換性を維持し、backendを先に公開する。frontendの配布が失敗しても、正常に稼働しているbackendを連動して自動rollbackしない。

### 1.6 Terraform Plan PRコメント

terraform-plan.yamlでPRにplan結果をコメントする際は、hashicorp/setup-terraform + github-scriptを使用する。

| ツール/Action | 説明 |
|--------------|------|
| hashicorp/setup-terraform | Terraform公式Action |
| actions/github-script | `terraform show`の出力をPRにコメント |

### 1.7 承認プロセス

terraform-apply.yaml実行前の承認プロセスは設定しない（ポートフォリオ用途のため）。frontendの通常デプロイとrollbackはGitHubの`production` environmentを使用して履歴を残すが、承認は必須化しない。

本番運用で承認プロセスが必要な場合は、GitHubの「Environment protection rules」を使用して、特定ユーザーの承認を必須にできる。

### 1.8 frontend-deploy.yaml 詳細設定

| 項目 | 設定値 |
|------|--------|
| ビルド実行箇所 | `ci.yaml`のfrontend検証job（`pnpm run build`） |
| 配布artifact | `frontend-dist`（main pushの成功候補を90日保持） |
| 配布方式 | immutable assetを先にS3へ配置し、短期キャッシュ対象JSON、`index.html`の順に公開 |
| 旧asset | rollbackと既存クライアント参照を保護するため通常配布では削除しない |
| Rollback | `frontend-rollback.yaml`に成功済みCI Workflowの`source_run_id`を指定してartifactを再配布 |

### 1.9 backend-deploy.yaml 詳細設定

| 項目 | 設定値 |
|------|--------|
| Dockerイメージのタグ付け | コミットハッシュ（`${{ github.sha }}`）を使用 |
| ECSデプロイ完了の待機処理 | あり（`aws ecs wait services-stable`を使用） |
| ロールバック | ECS deployment circuit breakerによるbackendデプロイ失敗時の自動rollback |
